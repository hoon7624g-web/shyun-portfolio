01

왜 최신 Realtime 대신 2초 폴링을 백업으로 남겼나?

Context

현장은 Wi-Fi 품질이 일정하지 않습니다. WebSocket이 수 초 끊기면 라운드 오픈·투표 마감 같은 타이밍 크리티컬한 순간에 UI가 stale해지고, 이는 현장 운영 자체의 붕괴로 직결됩니다.

Decision

Realtime을 Primary, 2초 간격의 얇은 폴링을 Fallback으로 병행. 네트워크가 깜빡여도 최대 2초 안에 상태가 복구됩니다.

Trade-off

서버 부하 미세 증가(Edge Functions 무료 티어 내 수렴) ↔ 화면 stale 위험 제거.

Architecture & Challenges

행사 특성상 하나의 이벤트에서 50~100명이 동시에 같은 RPC를 호출하는 순간이 있습니다. 라운드 오픈·투표 마감·매칭 결과 공개 — 초 단위 타이밍에 서버가 견뎌야 하고, 네트워크가 깜빡여도 화면이 stale 상태로 멈추지 않아야 했습니다. 일부 시나리오에서 Supabase Realtime의 안정성이 행사 운영 요구치에 미치지 못해, 핵심 화면(매칭 결과 공개·라이브 운영 화면 등)은 2초 폴링을 1차 채널로 두고 Realtime을 보조로 두는 hybrid 구조로 전환했습니다. 재접속 시에는 SHA-256 해시 기반 resume_token으로 세션을 복구하고, 동시 RPC 호출에는 ON CONFLICT 멱등 처리와 advisory lock을 조합해 race condition을 방어합니다.

Operations-First Engineering

  • 관리자 기능별 ON/OFF 제어 — 행사 도중 특정 기능만 즉시 끌 수 있어야 함. 어드민 패널 RPC 레벨 토글.
  • bcrypt 기반 어드민 인증 + 레이트 리밋 — 10분 내 10회 실패 시 자동 락, 공격 로그 1시간 자동 정리.
  • D+1 운영 반영 사이클 — 행사 현장 피드백을 다음 행사 전까지 닫는 운영 모델. 빠른 릴리즈 그 자체가 아니라, 피드백 → 분석 → 반영 → 검증 루프의 길이를 짧게 유지하는 데 초점.

PL/pgSQL 
-- 참가자 등록 · 재접속 시 idempotent 동작
-- resume_token 평문은 저장하지 않고 SHA-256 해시만 보관
create or replace function public.register_participant(
  p_room_code     text,
  p_user_id       uuid,
  p_display_name  text,
  p_resume_token  text
) returns jsonb
language plpgsql security definer as $$
declare
  v_participant_id uuid;
  v_token_hash     text;
begin
  v_token_hash := encode(digest(p_resume_token, 'sha256'), 'hex');

  insert into participants (room_code, user_id, display_name, resume_token_hash)
  values (p_room_code, p_user_id, p_display_name, v_token_hash)
  on conflict (room_code, user_id) do update
    set display_name      = excluded.display_name,
        resume_token_hash = excluded.resume_token_hash
  returning id into v_participant_id;

  return jsonb_build_object(
    'participant_id', v_participant_id,
    'status',         'ok'
  );
end; $$;

Multi-LLM Orchestration

각 모델의 강점에 맞춰 역할을 분배하는 게 파이프라인의 핵심입니다. Gemini는 긴 영상 입력과 멀티모달 분석에, Claude는 한국어 문장의 자연스러움에, GPT는 구조화된 키워드 추출에, Perplexity는 최신 팩트 체크에 각각 특화됩니다. 하나의 모델로 모든 것을 하려면 품질이 중간에 머물지만, 모델을 조합하면 각 단계에서 최고 성능을 낼 수 있습니다.

// 단계별로 강점이 다른 LLM에 라우팅 — hallucination 최소화
type Stage = 'analyze' | 'draft' | 'structure' | 'verify';

const ROUTE: Record<Stage, LLMProvider> = {
  analyze:   'gemini',      // 긴 영상 입력 · 멀티모달 분석
  draft:     'claude',      // 한국어 문장 톤
  structure: 'gpt',         // JSON 스키마 안정성
  verify:    'perplexity',  // 최신 팩트 체크
};

export async function runStage<T>(
  stage:  Stage,
  input:  StageInput
): Promise<T> {
  const llm    = clients[ROUTE[stage]];
  const result = await llm.invoke(input, {
    timeout: STAGE_TIMEOUT[stage],
    retries: 2,
  });
  return validate(stage, result);
}

Why a Supabase Edge Functions Proxy?

데스크톱 앱은 코드가 클라이언트에 통째로 배포되기 때문에, API 키를 앱 내부에 심으면 리버스 엔지니어링으로 탈취 가능합니다. Edge Functions를 프록시로 두면 사용자는 인증 토큰만 받고, 실제 Apify·LLM·YouTube API 호출은 서버에서 대신 수행합니다. 전형적인 BFF(Backend for Frontend) 패턴을 데스크톱 앱에 적용한 구조입니다.

Why Apify?

Instagram·TikTok은 공식 API가 극히 제한적이거나 상업용 크롤링을 금지합니다. Apify는 관리형 크롤링 서비스로 구현·운영 복잡도를 낮추고, apify~instagram-post-scraper와 clockworks~free-tiktok-scraper 같은 actor들을 바로 조합할 수 있습니다. free-tier actor의 가용성 리스크는 paid actor로의 fallback 경로를 미리 설정해 흡수합니다.

supabase/functions/apify-proxy/index.ts
TypeScript · Deno 
// 클라이언트 JWT만 검증 · API 키는 서버 env에만 존재
const APIFY_TOKEN = Deno.env.get('APIFY_TOKEN')!;

Deno.serve(async (req) => {
  // 1. 유저 JWT 검증
  const jwt = req.headers.get('Authorization');
  const { data: { user } } = await supabase.auth.getUser(jwt);
  if (!user) return new Response('Unauthorized', { status: 401 });

  // 2. 레이트 리밋 · 쿼터 제어
  if (await isRateLimited(user.id)) {
    return new Response('Too Many Requests', { status: 429 });
  }

  // 3. 프록시 호출 — 키는 서버에서만 주입
  const { endpoint, payload } = await req.json();
  const res = await fetch(`https://api.apify.com/v2/${endpoint}`, {
    method:  'POST',
    headers: {
      'Authorization': `Bearer ${APIFY_TOKEN}`,
      'Content-Type':  'application/json',
    },
    body:    JSON.stringify(payload),
  });
  return new Response(await res.text(), { status: res.status });
});

04 Craft

모델·언어·사람을
이어내는 법

LLM 기반 실행력과 비개발자 커뮤니케이션은 분리된 두 역량이 아닙니다. 모델의 출력을 사람의 맥락에 맞춰 번역하고, 사람의 요구를 모델이 다룰 수 있는 형태로 재구조화하는 — 하나의 동일한 근력입니다.

01 · Reliability

운영되는 서비스까지

세 프로덕트를 기획·구현·배포까지 단독으로 완결했고, 그중 하나는 실서비스로 직접 운영하며 로그·에러·사용자 피드백을 하루 단위로 반영하고 있습니다. 실행 속도와 안정성을 함께 챙깁니다.

02 · Global

영어 기반 실무 감각

TOEIC 970점(2025.06)의 비즈니스 영어 레벨을 유지하며, 해외 기술 문서와 글로벌 AI 동향을 원문으로 추적합니다. 한화오션에서 독일 제조사와 영어로 핫라인을 운영하며 생산설계 이슈를 실시간 조율한 경험이 있습니다.

기획·설계·개발·배포·운영까지 단독 수행한 경험, 그리고 내 앞의 사람이 이해할 수 있는 언어로 다시 짜는 감각을 동시에 가진 조합이 지금 가장 희소합니다. 한화오션 국방과학연구소 보고 자료, 대치동 수백 명의 학부모 대면 컨설팅, 학원 집합 강의 — 포맷은 달라도 일의 본질은 하나였습니다.

특히 잘 맞는 환경
  • AI를 조직에 실무형으로 도입·교육해야 하는 팀 — 실제 프로덕트를 만들어본 감각으로 현업형 교육 설계
  • 고객 현장과 제품 사이를 직접 오가는 역할 — 대면 컨설팅 4년으로 쌓은 고객 맥락 번역 능력
  • 콘텐츠·운영 자동화 환경을 직접 구축해야 하는 조직 — 멀티 LLM 파이프라인과 노코드 워크플로우를 실제로 엮어 반복 업무 시간을 단축한 경험
  • 빠르게 프로토타입을 만들어야 하는 팀 — 0→1을 단독 수행한 경험
  • 비개발자와 기술팀 사이 번역이 중요한 팀 — 방산·보고·컨설팅에서 쌓인 전달 감각
